# 物资入库与发票管理系统设计

## 1. 文档目的

本文档定义系统第一版设计，用于统一网页上传、文件落盘、Agent 定期审核、管理员手动入库确认、库内汇总、报销提取、发票重复校验这几条主流程。后续功能扩展以本文档为基础持续补充。

## 2. 系统目标

系统目标分为六部分：

1. 团队成员通过网页上传物资购买表格。
2. 服务端将上传内容按规则保存到未入库发票表文件夹。
3. Agent 定期审核未入库发票表文件夹中的分散表。
4. Agent 审核通过后将批次转入待入库发票表文件夹，等待管理员人工复核。
5. 管理员确认入库后，系统将分散表汇总进入库内总表，并将文件转入库内发票表文件夹。
6. 管理员提取库内数据用于报销后，系统将对应记录转入出库发票表文件夹，用作历史发票记录和重复报销排查依据。

## 3. 当前范围

第一阶段只处理以下能力：

- 网页端上传表格文件
- 四类本地文件夹管理
- Agent 定时审核
- 管理员手动入库确认
- 库内总表汇总
- 报销提取和出库归档
- 发票重复校验
- 汇总结果留痕
- 异常文件分流

第一阶段先处理基础报销流转，不处理复杂审批、多级权限、外部 ERP 对接。

## 4. 角色定义

- 团队成员：上传表格
- 管理人员：查看审核结果、执行人工入库确认、提取库内总表、登记报销出库
- Agent：执行文件扫描、解析、校验、去重审核、状态流转
- 系统服务：提供网页、接口、文件存储、数据库访问

## 5. 总体流程

```mermaid
flowchart LR
    A[团队成员上传表格] --> B[服务端写入未入库文件夹]
    B --> C[Agent 定时扫描未入库文件夹]
    C --> D[解析表格数据]
    D --> E[标准化字段]
    E --> F[检查发票重复]
    F --> G[审核通过转入待入库文件夹]
    G --> H[管理员手动确认入库]
    H --> I[汇总写入库内总表]
    I --> J[移动到库内文件夹]
    J --> K[管理员提取报销批次]
    K --> L[移动到出库文件夹]
    F --> M[写入异常记录]
    M --> N[移动到异常归档]
```

## 6. 目录结构设计

建议在系统根目录下使用固定目录结构：

```text
storage/
  invoices/
    unregistered/             # 未入库发票表文件夹，成员刚上传完成
    pending_review/           # 待入库发票表文件夹，Agent 审核通过待人工复核
    in_stock/                 # 库内发票表文件夹，管理员确认入库后待报销
    out_stock/                # 出库发票表文件夹，已完成报销
    processing/               # Agent 正在处理的文件
  archive/
    duplicate_invoice/        # 发票重复文件
    parse_failed/             # 表格解析失败文件
    validation_failed/        # 字段校验失败文件
    review_rejected/          # 管理员复核驳回文件
  master/
    in_stock_master/          # 库内总表导出
    out_stock_master/         # 出库历史总表导出
    reimbursement_export/     # 报销提取结果
  temp/                       # 临时处理目录
```

单次上传建议生成独立批次目录：

```text
storage/invoices/unregistered/{batch_id}/
  form.xlsx
  meta.json
```

`batch_id` 定义为上传批次唯一标识，建议格式为 `YYYYMMDDHHMMSS_提交人ID_随机串`。

四类主目录的业务含义如下：

- `unregistered`：远端上传完成，尚未经过 Agent 审核
- `pending_review`：Agent 审核通过，等待管理员人工确认是否正式入库
- `in_stock`：管理员确认入库完成，文件已进入库内管理，可打印报销
- `out_stock`：管理员完成报销提取，文件已作为历史记录封存

## 7. 上传批次结构

每次上传生成一个批次，批次包含三类信息：

- 表格文件：物资购买明细表
- 元数据文件：记录提交人、团队、提交时间、原始文件名、客户端备注
- 表格内发票字段：记录发票号码、发票日期、发票金额等审核信息

`meta.json` 建议字段：

```json
{
  "batch_id": "20260618163000_u001_a8f3",
  "team_name": "项目组A",
  "submitter_name": "张三",
  "submitter_id": "u001",
  "submitted_at": "2026-06-18T16:30:00+08:00",
  "form_file": "form.xlsx",
  "remark": "6月采购入库",
  "folder_stage": "unregistered"
}
```

## 8. 网页端设计

网页端先做单页面上传入口，页面包含以下字段：

- 团队名称
- 提交人姓名
- 提交人编号
- 采购表格上传
- 备注
- 提交按钮

网页端执行逻辑：

1. 用户填写提交信息并选择文件。
2. 服务端校验文件类型、大小、必填项。
3. 服务端生成 `batch_id`。
4. 服务端在 `unregistered` 下创建批次目录并写入文件。
5. 服务端返回上传成功结果和批次编号。

文件格式约束建议：

- 表格：`.xlsx`
- 单批次文件总大小：第一版可先限制为 50MB

## 9. 表格模板设计

第一版固定使用单一上传模板。模板固定会直接约束解析字段和审核规则，因此第一版不支持多模板混用。表格至少包含以下列：

| 字段名 | 含义 | 是否必填 |
| --- | --- | --- |
| 采购日期 | 发生采购的日期 | 是 |
| 物资名称 | 实际购买的物资名称 | 是 |
| 规格型号 | 物资规格 | 否 |
| 单位 | 数量单位 | 是 |
| 数量 | 采购数量 | 是 |
| 单价 | 单件价格 | 是 |
| 金额 | 行项目金额 | 是 |
| 供应商 | 销售方名称 | 否 |
| 发票代码 | 发票代码 | 否 |
| 发票号码 | 发票唯一识别号码 | 是 |
| 发票日期 | 开票日期 | 否 |
| 发票金额 | 发票总金额 | 否 |
| 备注 | 补充说明 | 否 |

系统处理逻辑建立在模板稳定的前提上，模板稳定会直接降低解析复杂度并提升汇总准确率。

## 11. Agent 处理流程

Agent 定义为按固定周期执行的后台任务。建议每 5 分钟扫描一次 `storage/invoices/unregistered/`。

单批次处理流程：

1. 读取未入库批次目录。
2. 将批次目录移动到 `processing/`，避免重复拾取。
3. 读取 `meta.json` 和表格文件。
4. 解析表格行数据。
5. 执行字段标准化。
6. 执行必填校验和数值校验。
7. 执行发票重复校验。
8. 写入审核结果和待入库记录。
9. 将批次目录移动到 `pending_review/`。
10. 记录处理日志。
11. 异常批次移动到异常归档目录。

字段标准化建议：

- 去除字符串首尾空格
- 统一日期格式为 `YYYY-MM-DD`
- 数量、单价、金额统一转为数值
- 发票代码、发票号码统一去空格并保留原始字符
- 发票字段按表格列直接读取

管理员手动入库流程：

1. 管理员查看 `pending_review` 中的待确认批次。
2. 管理员可按整批确认，也可按明细逐行确认。
3. 系统将确认通过的明细汇总写入库内总表。
4. 整批确认时，系统将批次目录移动到 `in_stock/`。
5. 逐行确认时，系统按明细状态更新批次和处理日志。
6. 未通过确认的内容直接打回异常目录。

管理员报销出库流程：

1. 管理员从库内记录中选择本次需要报销的批次或明细。
2. 系统生成报销导出文件。
3. 系统登记出库批次和出库明细。
4. 整批出库时，系统将对应文件从 `in_stock/` 移动到 `out_stock/`。
5. 明细出库时，系统按明细状态登记出库结果。
6. 系统将出库记录纳入重复报销排查范围。

## 12. 发票重复校验规则

发票重复检查是系统核心规则，第一版按以下顺序执行：

1. 生成发票去重键。
2. 在 `invoice_registry` 中查询是否已存在相同去重键。
3. 在 `reimbursement_record` 中查询是否存在已出库报销记录。
4. 命中重复键时，将当前批次标记为重复发票异常。
5. 命中已报销记录时，将当前批次标记为重复报销风险异常。
6. 未命中异常时，允许进入待入库阶段。

发票去重键建议定义：

```text
dedupe_key = invoice_number
```

第一版重复校验只按发票号码执行。发票号码 = 当前业务中的唯一核验字段 → 审核规则只使用 `invoice_number`。

发票与物资明细关系固定为一对一。一张发票对应一条物资明细，一条物资明细对应一张发票。

重复发票命中后建议记录以下信息：

- 当前批次编号
- 当前提交人
- 当前表格行号
- 已存在批次编号
- 已存在采购记录编号
- 发票代码
- 发票号码

重复报销排查建立在 `reimbursement_record` 持续积累的基础上。出库历史完整会直接提升后续报销审核准确率。

## 13. 批次状态设计

`upload_batch.status` 建议使用以下状态：

- `unregistered`：已上传到未入库文件夹
- `processing`：Agent 审核处理中
- `pending_review`：Agent 审核通过，待管理员确认入库
- `in_stock`：管理员确认入库完成
- `out_stock`：已提取报销并转入出库
- `duplicate_invoice`：命中重复发票
- `duplicate_reimbursement`：命中重复报销风险
- `parse_failed`：表格解析失败
- `validation_failed`：字段校验失败
- `review_rejected`：管理员复核驳回
- `db_failed`：数据库写入失败

状态流转稳定会直接支撑后续管理页面、重试任务、异常定位。

## 14. 异常处理设计

异常按类型分流：

- 表格打不开：进入 `parse_failed`
- 缺少必填列：进入 `validation_failed`
- 某一行发票重复：进入 `duplicate_invoice`
- 某一行发票已经完成报销：进入 `duplicate_reimbursement`
- 管理员复核不通过：进入 `review_rejected`
- 数据库写入失败：进入 `db_failed`

管理员复核不通过时先直接打回。直接打回 = 停止当前流转 → 移入异常目录 → 不提供在线修改后重新入库。

第一版建议采用批次级失败策略。批次级失败定义为批次内任一关键错误出现后，该批次整体停止流转并进入异常目录。批次级失败会直接保证总数据库状态一致。

## 15. 归档与可追溯性

系统需要保证每条汇总记录都可追溯到原始上传文件，因此需要同时保存：

- 原始表格路径
- 上传批次编号
- 表格行号
- 提交人信息
- 处理日志
- 当前文件夹阶段
- 入库确认人
- 出库报销批次

这条追溯链建立后，后续核查发票来源、定位异常数据、恢复处理过程都会更直接。

## 16. 第一版技术方案建议

第一版建议采用以下组合：

- 网页端：简单表单页面
- 后端接口：提供上传接口和文件保存逻辑
- 文件存储：本地文件夹
- 总数据库：SQLite
- Agent：定时审核任务脚本
- 管理端：网页管理页面，提供处理状态查看、汇总结果查看、待入库确认、报销提取

这套方案的核心价值是先形成完整闭环。闭环建立后，后续可以继续扩展用户登录、管理后台、异常复核页面、自动导出报表。

## 18. 已确认规则

当前版本已确认以下规则：

1. 上传表格模板固定为单一格式。
2. 发票信息通过表格字段获取，不提交发票文件。
3. 发票重复规则只按发票号码校验。
4. 发票与物资明细关系固定为一对一。
5. 管理员手动入库支持整批确认和逐行确认。
6. 报销提取支持整批出库和按明细出库。
7. 异常批次当前策略为直接打回。
8. 管理人员需要网页查看处理状态和汇总结果。
9. 总数据库第一版采用 SQLite。
